### 第一层：外部上下文

四个模块并列，负责从外部世界获取原始数据。

#### 1.1 USER/ — 用户品味语料

> "让用户真正属于他的那几个文件"

| 文件 | 说明 |
|------|------|
| `taste.md` | 用户音乐品味描述（喜欢的艺人、风格、年代偏好等） |
| `routines.md` | 日常作息规律（什么时间段做什么事、想听什么类型音乐） |
| `playlists.json` | 用户自定义歌单数据 |
| `mood-rules.md` | 情绪-音乐映射规则（什么心情对应什么 BPM/风格） |

#### 1.2 BRAIN — Claude Code

> "子进程调用 · 用 Max 订阅无限 API key"

| 项目 | 详情 |
|------|------|
| 调用方式 | `claude -p`（CLI 命令）通过子进程调用 |
| 输出格式 | `output.json`（结构化 JSON） |
| 作用 | 作为系统的 AI 大脑，负责自然语言理解、音乐推荐推理、播报文案生成 |
| 订阅方式 | Max 订阅计划（提供额度较高的 API 调用配额） |

#### 1.3 MUSIC — NeteaseCloudMusicApi

> "歌曲检索 · 直链 · 歌词 · 推荐"

| 端点 | 功能 |
|------|------|
| `search` | 歌曲搜索（按关键词、艺人、专辑名检索） |
| `song_url` | 获取歌曲播放直链（用于 PWA 播放器） |
| `lyric` | 获取歌词数据 |
| `recommend` | 获取个性化推荐（基于网易云算法） |

#### 1.4 VOICE·I/O — Fish·Feishu·Weather·UPnP

> "声音 · 日程 · 天气 · 客厅功放"

| 子模块 | 功能 | 接口/标识 |
|--------|------|----------|
| Fish Audio | TTS 文字转语音，生成 DJ 播报音频 | `fish tts` |
| 飞书 (Lark) | 读取用户日历/日程，获取当日安排 | `lark` |
| OpenWeather | 获取实时天气数据，注入推荐上下文 | `weather` |
| UPnP | 将音乐推送到局域网家庭音响播放 | `Nain`（可能为内部代号） |

### 第二层：本地大脑

> 六个 Node.js 模块，命名规则：`*_3S`（高频调度模块）/ `*_08`（持久化模块），是系统的核心调度与推理中枢。

#### 模块总览

| 序号 | 模块名 | 中文名 | 核心职责 | 关键行为 |
|------|--------|--------|----------|----------|
| 1 | **ROUTER_3S** | 意图分流 | 路由分发 | 简单指令直连 · 音乐走 ncm · 自然语言走 claude |
| 2 | **CONTEXT_3S** | 提示词组装 | 上下文拼接 | taste + routines + 环境·历史 → system prompt |
| 3 | **CLAUDI_3S** | 大脑适配器 | AI 推理 | spawn 子进程 · 解析 `{say, play, reason, segue}` |
| 4 | **SCHEDULER_3S** | 节奏调度 | 时间驱动 | 07:00 规划 · 09:00 早间 · 每小时情绪检查 · 日历 hook |
| 5 | **TTS_3S** | 声音管线 | 语音合成 | Fish Audio → `cache/tts/*.mp3` → `/tts/#hash-.mp3` |
| 6 | **STATE_08** | 状态·记忆 | 持久化 | messages · plays · plan · prefs → 跨重启持久 |

#### 2.1 ROUTER_3S — 意图分流

**职责**：接收第一层传入的各类数据和用户请求，判断意图类型并分发给正确的下游模块。

**路由规则**：

```
用户请求
  │
  ├─ 简单指令（如"播放/暂停/下一首/音量"）
  │   └─ 直连处理，不走 AI
  │
  ├─ 音乐类请求（如"搜某首歌/播放某歌单"）
  │   └─ 走 ncm（NeteaseCloudMusicApi）
  │
  └─ 自然语言请求（如"推荐适合雨天听的歌"）
      └─ 走 Claude Code（子进程推理）
```

**设计要点**：

- 快速路径：简单指令避免 AI 推理延迟
- 音乐路径：直接调用网易云 API，响应快速
- AI 路径：仅自然语言理解/推荐等需要推理的场景才走 Claude

#### 2.2 CONTEXT_3S — 提示词组装

**职责**：将所有上下文碎片拼接为完整的 system prompt，供 CLAUDI_3S 使用。

**拼接来源**：

| 来源类型 | 具体内容 | 文件/接口 |
|----------|----------|----------|
| 用户品味 | 音乐偏好、风格倾向 | `taste.md` |
| 作息规律 | 时间-场景-音乐映射 | `routines.md` |
| 环境信息 | 天气、日期、时间 | weather API |
| 日程信息 | 当日日历事件 | 飞书 Calendar API |
| 历史记录 | 最近播放、偏好变化 | STATE_08 |

**输出**：组装后的完整 system prompt，包含角色设定（persona）、用户画像、当前上下文。

#### 2.3 CLAUDI_3S — 大脑适配器

**职责**：通过子进程调用 Claude Code CLI，传入组装好的 prompt，解析结构化输出。

**调用方式**：

```bash
claude -p "<system_prompt>" > output.json
```

**输出 JSON 结构**：

```json
{
  "say": "DJ 播报的语音文案（将送入 TTS 合成）",
  "play": {
    "song_id": "歌曲 ID",
    "song_name": "歌名",
    "artist": "艺人"
  },
  "reason": "推荐理由（简短，解释为什么选这首歌）",
  "segue": "转场衔接语（用于两首歌之间的过渡播报）"
}
```

**四个字段的含义**：

| 字段 | 含义 | 用途 |
|------|------|------|
| `say` | DJ 播报的完整语音文案 | 送入 TTS_3S 合成为音频播放 |
| `play` | 当前要播放的歌曲信息 | 送入 ncm 解析获取播放链接 |
| `reason` | 本条推荐的理由 | PWA 界面展示（为什么推荐这首歌） |
| `segue` | 转场/衔接语 | 两首歌之间播报，实现 DJ 般的流畅过渡 |

#### 2.4 SCHEDULER_3S — 节奏调度

**职责**：时间驱动的事件调度器，按预设节奏触发系统行为。

**调度时间线**：

```
07:00  规划当天的音乐策略（一天的计划）
09:00  早间音乐推送（起床后/通勤时段）
每整点  情绪检查（根据当前上下文判断是否需要调整音乐风格）
日历 hook  当日历事件开始时/结束前触发（如会议结束后播放放松音乐）
```

**设计要点**：

- 非实时触发：不依赖用户主动输入，系统主动推送
- 与飞书日历联动：读取日程后自动安排对应时间的音乐策略
- 情绪检查：每小时轮询当前上下文（天气变化、时间推进），动态调整推荐

#### 2.5 TTS_3S — 声音管线

**职责**：将 Claude 生成的 `say` 文本合成为 MP3 音频，并提供 HTTP 访问端点。

**处理流程**：

```
say 文本 → Fish Audio API → cache/tts/{hash}.mp3 → /tts/{hash}.mp3
```

**缓存策略**：

- 以文本 hash 作为文件名，相同播报文案不重复合成
- 存储于 `cache/tts/` 目录
- 通过 HTTP 端点 `/tts/{hash}.mp3` 对外提供访问

#### 2.6 STATE_08 — 状态·记忆

**职责**：系统状态的持久化层，跨重启保持数据。

**存储内容**：

| 数据项 | 说明 |
|--------|------|
| `messages` | 用户与 Claudio 的历史对话记录 |
| `plays` | 播放历史（时间、歌曲、是否跳过/收藏） |
| `plan` | 当日/当周的音乐策略计划 |
| `prefs` | 用户偏好（随着使用逐渐学习的动态画像） |

**持久化方式**：`state.db`（推测为 SQLite 或 JSON 文件），本地存储，数据完全归属用户。

#### 第二层数据流（管道顺序）

```
第一层数据输入
    │
    ▼
ROUTER_3S ──── 判断意图，分发给不同路径
    │
    ├─ 简单指令 → 直接处理返回
    ├─ 音乐请求 → ncm API → 返回播放链接
    │
    └─ 自然语言请求
          │
          ▼
    CONTEXT_3S ──── 拼接完整 system prompt
          │
          ▼
    CLAUDI_3S ──── spawn 子进程调用 Claude，解析 JSON
          │
          ├─ say ──────→ TTS_3S ──→ 合成语音
          ├─ play ─────→ ncm ────→ 获取播放链接
          └─ reason/segue → PWA 界面展示
          │
          ▼
    STATE_08 ──── 持久化本轮所有状态变更
```

### 第三层：运行时聚合

> 核心机制：每次触发时，将 6 片「碎片」粘合成一个 CONTEXT WINDOW 送入模型。

#### 6 片碎片详解

| 序号 | 碎片名称 | 来源 | 说明 |
|------|----------|------|------|
| ① | 系统提示词 | `prompt/@-persona.md` | Claudio 的角色设定、播报风格、行为准则 |
| ② | 用户语料 | `user/*.md` | taste.md / routines.md / playlists.json / mood-rules.md |
| ③ | 环境注入 | `weather` · `calendar` · `nwm` | 实时天气、当日日程、其他环境变量 |
| ④ | 已检索记忆 | `state.db` · `plays` | 历史播放记录、偏好演化数据 |
| ⑤ | 用户输入 / 工具结果 | `/api/chat` · `ncm search` | 用户本轮输入 + 音乐搜索结果 |
| ⑥ | 执行轨迹 | `scheduler` · `webhook` | 当前调度上下文、触发原因 |

#### 前向推理流程

```
┌─────────────────────────────────────────┐
│          碎片 ①~⑥ 按优先级排列           │
│         组装为 CONTEXT WINDOW            │
└─────────────────┬───────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────┐
│     Claude 模型推理                      │
│     输出: {say, play, reason, segue}     │
└─────────────────┬───────────────────────┘
                  │
       ┌──────────┼──────────┐
       ▼          ▼          ▼
   say 文本    play 歌曲   reason/segue
       │          │           │
       ▼          ▼           ▼
   TTS 合成   ncm 解析     PWA 展示
       │          │
       ▼          ▼
   语音播出    播放器播放
       │
       ▼
  WS 推 now-playing 到 PWA
```

### 第四层：交互界面

#### 技术栈

| 项目 | 详情 |
|------|------|
| 技术方案 | PWA（Progressive Web App） |
| 运行地址 | `LOCALHOST:8080` |
| 三视图 | Player（播放器）/ Profile（个人资料）/ Settings（设置） |
| 优化策略 | 存壳层 + prefetch 10s（预加载策略，缓存壳资源，提前 10 秒预取下一首） |

#### HTTP 通信契约（PWA ↔ Server）

| 序号 | 端点 | 方法 | 方向 | 说明 |
|------|------|------|------|------|
| 1 | `/api/now` | GET | PWA → Server | 获取当前播放状态（歌名、艺人、进度） |
| 2 | `/api/taste` | GET | PWA → Server | 获取/更新用户品味数据 |
| 3 | `/api/plan/today` | GET | PWA → Server | 获取今日音乐策略计划 |
| 4–6 | `/stream` | WS | Server → PWA | WebSocket 双向通道，实时推送 now-playing 状态、语音播报进度、歌曲切换通知 |

> 注：图中标注"6 条线"，实际端点 4 个，其中 WebSocket 连接可能包含多条逻辑通道（播放状态、语音进度、通知）合并计为 6。

#### PWA 特性

- **离线可用**：壳资源（HTML/JS/CSS）缓存，无网络时仍可打开基础界面
- **类原生体验**：可添加到手机主屏幕，全屏运行
- **Prefetch 策略**：提前 10 秒预加载下一首歌曲的音频和 cover 图

#### 绿色虚线箭头（特殊连线）

架构图第四层左侧有一条绿色虚线箭头，标注「发给 Claude Code」：
- PWA 界面的截图可以反馈给 Claude Code
- 用于迭代开发：在 Claude Code 中看到当前界面效果后，自动生成 UI 改进建议
- 这是 Vibe Coding 理念的体现：用 AI 辅助 AI 项目的开发

## 三、所需 API Key 与权限汇总

| API / 服务 | 用途 | 所需凭证 | 是否必需 |
|------------|------|----------|----------|
| **Claude API** (Anthropic) | AI 大脑推理 | API Key（Max 订阅计划） | ✅ 必需（核心） |
| **NeteaseCloudMusicApi** | 歌曲检索、直链、歌词、推荐 | 网易云音乐 API 权限 | ✅ 必需（音乐源） |
| **Fish Audio** | TTS 语音合成 | API Key | ✅ 必需（语音播报） |
| **飞书 (Lark) API** | 读取用户日程 (Calendar) | 飞书应用凭证 | ⚠️ 可选（无日程则跳过） |
| **OpenWeather** | 天气数据注入 | API Key | ⚠️ 可选（无天气则跳过环境注入） |
| **UPnP** | 局域网推音乐到家庭音响 | 局域网权限（无需外部 API） | ⚠️ 可选（仅本地播放也 OK） |

## 五、关键设计理念

### 5.1 本地优先 (Local-First)

- 整个系统运行在 `localhost`，核心服务部署在用户本地机器
- 用户数据（taste.md、routines.md、state.db）完全存储于本地
- 不依赖云端服务做核心逻辑，外部 API 仅提供辅助能力

### 5.2 子进程调用 Claude Code

- 不直接调用 Claude API，而是通过 CLI 工具 `claude -p` 子进程调用
- 利用了 Claude Code 已有的上下文管理能力
- 输出为结构化 JSON，便于程序解析和下游消费

### 5.3 碎片化上下文组装

- 将系统提示词、用户语料、环境信息等拆为独立碎片
- 运行时动态组装，支持热更新（修改 taste.md 即时生效）
- 每次推理都重新拼接，保证上下文始终最新

### 5.4 时间驱动 + 事件驱动

- 不依赖用户主动交互，SCHEDULER 按时间线主动推送
- 与飞书日历联动，实现场景感知（会议结束后自动切换音乐）
- 每小时情绪检查，动态调整推荐策略

### 5.5 Vibe Coding 开发模式

- 架构图直接截图给 Claude Code，让 AI 辅助开发
- "Claude 的施工图"本身就是给 Claude Code 看的开发文档
- 绿色虚线箭头体现了"用 AI 开发 AI 产品"的 Vibe Coding 理念

### 5.6 渐进式增强

- PWA 技术选择支持离线访问和类原生体验
- API 依赖可降级：无飞书日程 → 跳过日程注入，无天气 → 跳过环境注入
- 核心链路（Claude + 网易云 + Fish Audio）才构成最小可用系统
